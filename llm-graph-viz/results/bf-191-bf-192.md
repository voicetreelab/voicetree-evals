# Results — BF-191 + BF-192 (the completed row)

This document reproduces the one completed measurement run from the upstream voicetree empirical work, reframed as benchmark rows. It also reports fresh fidelity + byte-cost measurements on the three public fixtures shipped in `fixtures/`.

**What was NOT done yet.** The one axis of the design that this row does *not* cover is per-question LLM scoring (§4 of [`design.md`](../design.md)). A follow-up task is to run the 30-question set across formats A/B/C/E/G/H and the four frontier models. This row shipped only the format-fidelity and format-cost axes, which were already measured end-to-end in the upstream commits.

## 1. Upstream row — `voicetree-public/brain/knowledge/world-model`

Source commits in `voicetree-public`:

- `f7042f61` — *BF-191*: shipped format C (ASCII + `[Cross-Links]` footer) in `packages/graph-tools/src/viewGraph.ts`.
- `e9c2d49f` — *BF-192*: shipped tree-cover renderer + arboricity in `packages/graph-tools/scripts/L3-BF-192-tree-cover-render.ts`.

Fixture: internal zettelkasten vault, N=465 nodes, E=476 directed edges (non-self), arboricity a(G)=3. Nash-Williams LB = greedy UB = degeneracy = 3, all three bounds tight.

### 1.1 Fidelity + size per format (world-model)

| Format | Node fidelity | Edge fidelity | Edges co-located? | Bytes | ≈ Tokens | Token multiplier vs A |
|---|---:|---:|---|---:|---:|---:|
| A — ASCII lossy (`vt-graph view` legacy) | 97.4% (452/464)¹ | 79.4% (378/476) | ✅ inline | 98 039 | 24 509 | 1.0× |

| **C — ASCII + `[Cross-Links]` footer** (shipped, BF-191) | **97.4%** | **98.1% (raw)** / **100% (rendered-subgraph)** | partial — tree ✅, footer mean 420-line lookup | 318 414³ | 79 603 | 3.2× |

Notes:

1. The 12 missing nodes in A/C node fidelity are `.voicetree/prompts/*` hidden-path files that `vt-graph view` filters at scan time. Not a format-layer issue.
2. Mermaid's < 100% edge fidelity comes from `vt-graph view --mermaid` dropping unresolved-wikilink targets. A different mermaid renderer could reach 100% at a similar byte cost.
3. The C byte cost in BF-191's internal scripts includes the full `[Cross-Links]` footer output by `renderAscii`. The reduced 3.2× multiplier (vs. the raw ~1.8× estimate from the synthesis document) is because this row counts the full on-disk artifact including the legend section.

### 1.2 Co-location (format E, BF-192 measurement)

Per BF-192 `findings.md` §4: for format E (tree-cover), every rendered edge has its source title on the `●` line 1–N lines above (within one `●` block, N ≤ out-degree ≈ 1–4) and its target title on the same line as the edge marker `⇢`. Mean source-to-target line gap: **< 5 lines per edge**, worst case bounded by out-degree.

For comparison, format B (Mermaid) has mean source-to-target line gap of **957 lines** per edge on the same fixture (labels at the top of the file, edges at the bottom). Format C's footer lookup has a median **420-line** gap from any given tree line to the `[Cross-Links]` header.

### 1.3 Arboricity decomposition (format E)

Greedy decomposition on world-model yielded 3 forests — matching the Nash-Williams lower bound. Forest sizes:

| Cover forest | Edge count | Shape |
|---|---:|---|
| Forest 1 | 372 | Dense argument-graph spine |
| Forest 2 | 40 | Side-thread cross-references |
| Forest 3 | 3 | Long-tail cross-vault references |

The dense-then-sparse distribution is the BF-192 signature of a zettelkasten with *one dominant argument web* plus a handful of side-threads (see BF-192 decision.md for interpretation).

## 2. Public-fixture rows — reproduced on the shipped benchmark fixtures

These rows measure the public fixtures (`fixtures/tree-20.json`, `fixtures/kanban-100.json`, `fixtures/world-model-like-465.json`) through this repo's own renderers (`scripts/render.ts`). They are fully reproducible — `pnpm run generate-fixtures && pnpm render <fmt> <fixture>` regenerates them bit-for-bit.

### 2.1 Fixture summary

| Fixture | Nodes | Edges | Arboricity (LB / greedy UB) | Shape |
|---|---:|---:|---|---|
| `tree-20` | 20 | 5 | 1 / 1 | Small project tree with 5 cross-references |
| `kanban-100` | 100 | 152 | 6 / 11 | 12-node hub clique + sparse tail — adversarial for tree-cover |
| `world-model-like-465` | 465 | 483 | 3 / 3 | Zettelkasten shape calibrated to match BF-192 world-model |

**On the greedy UB gap for `kanban-100`.** The Nash-Williams LB is tight at 6 (K_12 hub clique has arboricity exactly ⌈12/2⌉ = 6), but the simple forest-by-forest greedy used in `scripts/arboricity.ts` overshoots to 11 on dense subgraphs. This is a known limitation of simple greedy vs. Gabow-Westbrook — it doesn't affect the lower bound's correctness, only the number of cover forests emitted by `renderTreeCover`. For benchmarking purposes the reported arboricity bucket uses the LB.

### 2.2 Byte cost + format-C fidelity

Measured via `wc -c` on the rendered files in `formats/`:

| Fixture | A (bytes) | B (bytes) | C (bytes) | E (bytes) | H (bytes) | C edge-fidelity | C node-fidelity |
|---|---:|---:|---:|---:|---:|---:|---:|
| `tree-20` | 929 | 1 719 | 1 374 | 1 961 | 2 755 | 100% (5/5) | 100% (20/20) |
| `kanban-100` | 15 995 | 16 519 | 26 092 | 26 769 | 29 912 | 100% (152/152) | 100% (100/100) |
| `world-model-like-465` | 49 828 | 105 827 | 93 590 | 98 670 | 130 663 | 100% (483/483) | 100% (465/465) |

Token approximations (bytes / 4) for `world-model-like-465`:

| Format | ≈ Tokens | Token multiplier vs A |
|---|---:|---:|
| A | 12 457 | 1.0× |
| B | 26 457 | 2.1× |
| C | 23 397 | 1.9× |
| E | 24 667 | 2.0× |
| H | 32 665 | 2.6× |

**Why the public-fixture E multiplier (2.0×) is lower than the real world-model (4.4×).** The real world-model has mean path-ID length of 233 characters — titles are long because note filenames encode the thesis they argue ("alignment-determines-human-welfare-if-agents-decide-ai-invention-is-retrieval-plus-minimum-disruption.md"). The synthetic `world-model-like-465` fixture uses much shorter IDs (`wm/economic-dynamics/sub-02/note-037`). This means format E's token cost is dominated by per-ID byte count, and the multiplier is highly corpus-specific. The short-ID optimization proposed in BF-192 decision.md §R2 would compress the real world-model toward the synthetic-fixture ratio.

## 3. What rows need to be added

The design in `design.md` calls for cells across `format × fixture × model × question`. This row fills the `format × fixture` surface for three formats (C shipped, E shipped, A/B/H derivable) and omits the `model × question` axes entirely.

Next steps, in priority order:

1. **Small-scale LLM run** on `tree-20` (cheapest fixture) with format E and Claude Opus 4.7 — 30 questions × 1 model × 1 format × 1 graph = 30 API calls. Establishes the rubric works and the format doesn't confuse the model.
2. **Cross-format run on `tree-20`** — same 30 questions across A, B, C, E, H. First format-spread measurement.
3. **Scale-up to `world-model-like-465`** once the scoring pipeline is debugged.
4. **Cross-model** — repeat on Sonnet 4.6, Gemini 2.5 Pro, GPT-5.

A naive cost estimate: at Claude Opus rates, the full `format × fixture × question × model` grid (5 formats × 3 fixtures × 30 questions × 4 models = 1 800 calls) at mean ~20 k input + 500 output tokens/call ≈ $450 one-shot. Low enough to just run.

## 4. Reproducing this row

```
# Regenerate the three fixtures
pnpm install
pnpm run generate-fixtures

# Render all five formats for all three fixtures
for f in tree-20 kanban-100 world-model-like-465; do
  for fmt in A B C E H; do
    pnpm render $fmt fixtures/$f.json > formats/$f/$fmt-*.txt
  done
done

# Verify 100% round-trip fidelity on format C
pnpm run parse score fixtures/world-model-like-465.json \
  formats/world-model-like-465/C-ascii-footer.txt

# Compute arboricity on each fixture
for f in tree-20 kanban-100 world-model-like-465; do
  pnpm run arboricity fixtures/$f.json
done
```
