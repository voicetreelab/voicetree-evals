## HANDOVER.md Open Questions — Answers from Live Runs

**Does `.run.json` expose per-call token breakdown or only aggregates?**
Both. Each `.run.json` has:
- `conversations[].requests[].metrics` — per-request: `inputTokens`, `outputTokens`, `inputTokensCostNanodollars`, `outputTokensCostNanodollars`, `totalBackendLatencyMs`
- `conversations[].metrics` — per-conversation aggregate (same as request totals when there is 1 request)

In this single-call HCH protocol, per-request == per-conversation. For multi-call variants, the per-request breakdown would be informative for per-subtask token accounting.

**Can HCH recursion fit cleanly inside one `@kbench.task`?**
YES, confirmed by both runs. A single `@kbench.task(fn(llm) -> bool)` with one `llm.prompt(PROMPT)` call handles the full PLAN/EXECUTE/INTEGRATE trajectory. Parsing the response extracts subtask plans, execution markers, and final answer. Both q1 and q2 used this pattern successfully. The fresh-session 3-call variant (one call per phase) would also fit inside one `@kbench.task` as sequential `llm.prompt()` calls — not tested in this spike.

**Does Kaggle's "Save Task" flow work cleanly for each pilot, and is the leaderboard view useful?**
Not yet confirmed — "Save Task" is a manual UI step in the Kaggle notebook after a successful run. Both tasks produced `.task.json` + `.run.json`, so the kernel side is complete. User needs to click "Save Task" in the Kaggle UI at the notebook URL for each task.

## Bugs Found and Fixed

**`from __future__ import annotations` breaks `@kbench.task` on the remote kernel.**
The `kaggle_benchmarks` `@kbench.task` decorator calls `_infer_result_type(func)` which inspects the return annotation. With `from __future__ import annotations` (PEP 563), annotations become strings at definition time (`'bool'` instead of `bool`). The remote kernel's `_infer_result_type` then fails: `TypeError: Return type annotation 'bool' ... is not a registered result type`. Fix: remove `from __future__ import annotations` from task files. This is a silent gotcha — the local smoke test (which only tests SDK import, not kernel execution) does NOT catch it.

Fix applied to: `examples/hch_spike/q1.py`, `examples/hch_spike/q2.py`.

## Known gotchas (new pilot-specific, beyond README section 7)

1. **`from __future__ import annotations` breaks `@kbench.task` on remote kernel.** See "Bugs Found" above. Remove this import from all task files.
2. **Default `idle_wait_seconds=10.0` is too short for LLM tasks.** The bridge polls the kernel state for 10s before raising `KernelBusyError`. If the kernel is processing an LLM call (takes 7–16s), a fast retry immediately after the previous run will fail. Fix: increase `idle_wait_seconds` to ≥20s when the kernel may be in post-execution cooldown, or wait 15–30s between bridge calls.
3. **`submit_task.py` default `timeout_seconds=30.0` may be insufficient for longer prompts.** Both q1 (7.4s) and q2 (15.6s) completed within 30s, but larger/more complex tasks could exceed this. For safety, call `bridge.run(remote_code, timeout_seconds=180.0)` directly.
