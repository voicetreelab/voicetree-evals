## Forecast vs Realized Calibration

- Gemini 3.1 Pro had the best plan-stage continuation calibration (`mean continue brier = 0.0013`) on the two seeds where the plan parsed. It was directionally right about “take one more subtask” both times. The weak spot was robustness: seed 3 died at plan parse, so the run never entered execution.
- Claude Sonnet 4.6 had decent thresholded calibration overall (`atomic brier = 0.0342`, `continue brier = 0.0216`) and better final-quality calibration than Gemini (`0.1981` vs `0.4342`). Its problem was cost and long-tail runtime, not obviously belief quality.
- GPT-5.4’s atomic forecast Brier looked superficially acceptable (`0.0366`), but its continuation forecasts were badly misaligned with realized value (`continue brier = 0.6297`). It repeatedly chose to continue even though every scored run fell back to baseline.

## One-Shot vs Multi-Turn Behavior

- Gemini 3.1 Pro behaved like the earlier `6x7` spike on the successful seeds: one planning turn, one execution turn, then stop. It did not explore deeper search, but the one-shot execution quality was strong when the plan parsed.
- Claude Sonnet 4.6 was the only model here that consistently explored beyond one execution step. Seed 2 used three execution subtasks; seed 3 used two completed execution subtasks before timing out on the next step.
- GPT-5.4 took the most execution turns on average, but none of them converted into a valid improving scored schedule. All three runs ended with `final_schedule_source=baseline`, with `2-3` feasibility failures per seed.

## Blockers and Anomalies

- The current Kaggle port has a timeout-fidelity gap: `kbench.llm.respond()` is blocking, so the `600s` subtask budget is only observable after return, not enforced as a hard kill from inside the task. Claude Sonnet seed 3 exceeded the intended limit and was converted into `subtask_timeout` by manually interrupting the Kaggle kernel. This matches the intended benchmark semantics, but it is not yet a faithful implementation of hard per-turn kill.
- The first Gemini smoke artifact accidentally wrote a literal `\\n` suffix instead of a real JSONL newline. That artifact was normalized into the merged file `cjs_5x6_google_gemini-3_1-pro-preview_20260416_full.jsonl`.
- GPT-5.4 was fast but structurally weak on this task family. The issue does not look like latency budget; it looks like repeated infeasible schedule proposals causing baseline fallback.

## Headline Takeaways

1. Gemini 3.1 Pro won this `5x6` spike on mean score, with one exact solve and one strong single-subtask improvement, but it had one brittle plan-parse failure.
2. Claude Sonnet 4.6 showed the best evidence of deeper search, but paid heavily in wall-clock and exposed the missing hard-timeout enforcement in the current Kaggle port.
3. GPT-5.4 did not produce a single scored improving schedule across the three seeds. It spent turns, but those turns mostly translated into infeasible candidates and baseline fallback.

## Appendix A: GPT-5.4 Infeasibility Root Cause

**Verdict: Real model behavior (machine overlap errors in schedule construction). Not a parser or format issue.**

Diagnosis methodology: inspected `turns[*].raw_text` and `turns[*].verification` for all 9 exec turns across seeds 1/2/3. Parser `parse_ok=True` on every turn — GPT-5.4 emitted valid `BEST_GUESS` JSON that the harness parsed and verified correctly. The verification step flagged each schedule as infeasible.

Specific failure pattern observed across all seeds and all exec turns:
- `verification.is_feasible = False` on every exec turn
- `verification.failure_reason` consistently reports **machine overlap** (e.g., `"machine overlap between J2 and J5"`, `"machine overlap between J5 and J3"`)
- GPT-5.4's Factory A schedule was notably repetitive across turns — the same MA1/MA2 block appearing verbatim in successive "repaired" attempts (seed 1, turns 2–4; seed 2, turns 2–4), indicating the model was patching a locally fixed template rather than reasoning from constraint satisfaction

Root cause: GPT-5.4 cannot reliably construct valid non-overlapping machine assignments for this 5×6 coupled problem. It narrates constraint satisfaction ("all machine capacities, within-job precedences respected") but the emitted intervals violate those constraints in every attempt. This is a **capability gap on this problem class**, not a harness bug.

What was ruled out:
- Parser bug: `parse_ok=True` on all exec turns; the harness correctly extracted and verified the schedule
- Format drift: GPT-5.4 followed the `BEST_GUESS: {...}` output protocol faithfully
- Tokenizer/stop-sequence: responses were complete and well-formed

Implication for benchmark design: GPT-5.4's fast wall_s (30–46s total per run) combined with 0% feasibility and a badly calibrated `CONTINUE_FORECAST` (`brier=0.63`) reveals that this model has no grounding signal about the infeasibility of its own outputs. A benchmark scoring rubric that penalizes continued execution after repeated infeasibility failures would surface this signal more clearly.

No re-run recommended: re-running with the same prompt and harness will reproduce the same infeasible outputs. A targeted fix would require either changing the prompt to provide a step-by-step constraint-checking template, or providing intermediate verification feedback to the model (which changes the protocol).

## Appendix B: Harness Fixes Applied (2026-04-16)

**Hard-kill on subtask budget** (`kaggle/examples/coupled_jobshop_spike/cjs_5x6.py`):
- Added `import threading` to imports
- `_call_model()` now wraps `llm.respond()` in a `threading.Thread(daemon=True)` with `.join(timeout=timeout_s)`
- If the thread is still alive after `timeout_s`, returns `timed_out=True` immediately without waiting — enforces budget as a true reject rather than measure-after-return
- Note: Python threading cannot forcibly SIGKILL a blocking network call; the daemon thread will complete in background. The harness advances to the next seed correctly.

**Plan-turn JSON fallback parser** (`kaggle/examples/coupled_jobshop_spike/cjs_5x6.py`):
- `parse_plan_turn()` now includes a fallback path for models that wrap the entire plan response in a code-fenced JSON blob with uppercase keys (e.g., `{"ATOMIC_FORECAST": ..., "DECISION": "continue", ...}`)
- Root cause of Gemini seed 3 `plan_parse_fail`: model emitted ` ```json\n{"ATOMIC_FORECAST": ...}\n``` ` instead of the expected plain-text label format. The `_extract_label_block` regex requires unquoted `LABEL:` at line start, which doesn't match JSON keys
- Fix: if label extraction returns `None` for any required field, `_parse_object_loose(text)` is tried on the full response text; uppercase and lowercase key variants are both checked
- Verified fix against captured Gemini seed 3 raw_text: parser now returns valid parsed dict
