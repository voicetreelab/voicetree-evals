### Axis C paradox
Axis C looks good (0.026 Brier) but is **misleading**: the model marked all subtasks as `correctly_solved=True` because it completed writing text for each subtask, even though the final answers were wrong. Sub-task "solved" means "I wrote something" not "the sub-answer is right." This is a design gap: `correctly_solved` needs external verification, not just model self-report.

## Headline Findings

1. **HCH did NOT beat vanilla**: 0% vs 0% official, 0% vs 16.7% LaTeX-corrected. Vanilla scored higher.

2. **Atomic baseline calibration is poor** (Axis A1 Brier = 0.537). The model was over-optimistic about its atomic accuracy on hard HLE questions. It predicted high `p_correct_if_atomic` on questions it ultimately got wrong.

3. **Decomposition never paid off** (Axis A3: 0/7 paid off). In every case where the model chose to decompose, it did NOT score higher than vanilla. For this model+question set, decomposition added latency and tokens without accuracy gain.

4. **Final-answer confidence is catastrophically overconfident** (Axis D Brier = 0.934). Model reported P_CORRECT ~0.9–1.0 on every answer, including provably wrong ones.

5. **Format non-compliance is the primary measurement blocker**: Gemini 2.5 Flash outputs `$\boxed{X}$` LaTeX notation by default for vanilla arm, defeating `ANSWER:` regex parsing. Also affected HCH arm (Q43, Q49, Q68 answer=None). This is the same bug documented in the MetaCoach pilot.

## Bugs Found

### Bug 1: Vanilla format non-compliance (LaTeX boxing) — CRITICAL
**Symptom:** All 12 vanilla answers parsed as `None` because model outputs `$\boxed{X}$` instead of `ANSWER: X`.
**Cause:** Gemini 2.5 Flash default output style. Same bug as MetaCoach pilot (2026-04-15).
**Fix options:**
  a. Add explicit format instruction: "Do NOT use LaTeX or boxed notation."
  b. Extend parser to also accept `$\boxed{X}$` format.
  c. Both.
**Impact:** All vanilla Axis D Brier scores are N/A; vanilla accuracy shows 0% instead of 16.7%.

### Bug 2: HCH arm format drift on long questions (Q43, Q49, Q55, Q68)
**Symptom:** `answer=None` parsed from HCH arm on 4/12 questions.
**Cause:** On long/complex questions, the model either:
  - Didn't emit `ANSWER:` at the end (Q43, Q68 — verified in raw response)
  - Produced a verbose essay instead of a crisp answer (Q55)
  - Hit context/response limits for complex questions (Q49: 2,158 output tokens, Q43: 3,163 tokens)
**Fix:** Add stronger integration step instruction, e.g., "The final line MUST be `ANSWER: <answer>` (plain text, no LaTeX)."

### Bug 3: f-string NameError for `flag{no_zeros}` — FIXED
**Symptom:** Q44 both arms crashed with `NameError: name 'no_zeros' is not defined`.
**Cause:** Gold answer `flag{no_zeros}` embedded in f-string as `gold='flag{no_zeros}'` — Python tries to evaluate `{no_zeros}` as a variable.
**Fix applied:** Changed to `gold='flag{{no_zeros}}'` to escape braces.
**Impact:** Q44 ran successfully on re-run. Model answered `flag{r_is_not_0}` (wrong) for HCH arm.

### Bug 4: Q49 + Q68 — protocol non-compliance
**Symptom:** n_subtasks=0, no SUB markers in HCH arm.
**Cause:** For Q49 (DFA minimization, complex regex) and Q68 (modal logic), the model appeared to drop the PLAN/EXECUTE protocol entirely and give a direct answer, parsing produced no SUBTASKS array or SUB blocks.
**Impact:** Axis A3/B/C data missing for these questions.

## Limitations

- n=12 questions — statistically non-representative
- Single model (`google/gemini-2.5-flash`)
- Single seed (no temperature variation)
- Format bugs substantially inflate apparent failure rate (esp. vanilla arm)
- "Correct" judgment for float Q41 and flag Q44 uses exact-match after parsing — could explain some edge failures
- HLE questions sampled from positions 41–99 of text-only filter; not a random sample of the benchmark's difficulty distribution
