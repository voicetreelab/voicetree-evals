# Port 1 masked_block_jobshop question into the Kaggle+local setup (spike)

You are an Opus investigator. Your mission: port **one single** masked_block_jobshop (MBJ) instance from the pilot-era codebase into the current benchmark shape, end-to-end, well enough that it lands in `questions.jsonl`, passes the canonical round-trip check, and is runnable by the local harness. If you succeed, this proves the port path is viable and unlocks a follow-up to add MBJ as a 7th class.

## Why this is a spike, not a full class port

Scope is deliberately narrow: **one instance** (medium difficulty, seed 1 is fine) through the full pipeline. Not 12 instances, not both difficulties, not portfolio integration. The goal is proof of path, not coverage. If the port uncovers a structural incompatibility, bail early and report the blocker — don't try to invent workarounds outside scope.

## Sources — pilot code that worked

- `hch/masked_block_jobshop/jobshop_instance.py` — the instance generator from the pilot spike
- `hch/masked_block_jobshop/run_spike.py` — pilot runner (reference only; do NOT port the runner, it's legacy)
- `hch/masked_block_jobshop/force_one_more_turn.py` — pilot-era turn logic (reference only)

MBJ is distinct from `cjs` (coupled jobshop), which is MBJ's descendant. MBJ masks blocks of operations rather than coupling two factories with shared machines. If after reading the pilot code you conclude MBJ is functionally identical to `cjs`, stop and report that — no point porting a duplicate.

## Target shape — what "works" looks like in the current setup

The current benchmark has a specific architecture you need to match. Read these exemplars first (point anchors, not deep reads):

- `kaggle_submission/generators/cjs.py` — exemplar generator: `generate(seed, difficulty)` returns dict; `build_instance` does full verification loop; dataclasses for spec; `DIFFICULTY_CONFIGS` with min_baseline_gap_pct
- `kaggle_submission/verifiers/cjs.py` — exemplar verifier: callable `(instance, submission) -> (score, feasible, info)`
- `kaggle_submission/verifiers/__init__.py` — `CLASS_TO_VERIFIER` registry; you add your class here
- `kaggle_submission/scripts/build_questions.py` — canonical row builder; add `_build_mbj_row` + a `SmokeRow` path; NOTE: `SOLO_CLASSES = ("cjs", ...)` should be extended only if your scope requires — for a one-row spike you can append manually in `_build_rows()` without touching `SOLO_CLASSES`
- `kaggle_submission/harness/prompt.py` — prompt template; check if per-class branching is needed for MBJ
- `kaggle_submission/harness/protocol.py` — parser; MBJ answer-contract needs parser support

## Acceptance criteria (what to verify)

1. `mbj_medium_seed1` row appears in `kaggle_submission/questions.jsonl` after running the build script.
2. `_sanity_check_round_trip` in `scripts/build_questions.py` passes for the new row (JSON round-trip + verifier-on-wrong-submission returns infeasible).
3. The harness can **construct the prompt** for the new row without crashing. Actually invoking an LLM is out-of-scope for this spike — you just need the prompt-render path to produce a valid string.
4. Optional stretch: run the row through the harness with a stub/mock LLM response to confirm protocol.py parses an MBJ answer. Skip if this adds > 30 min.

**Do NOT** run actual LLM calls. This is a generator/verifier/harness-plumbing spike, not an eval.
