# Phase 0 Sonnet Parse-Failure Diagnosis — Root Cause Found

All 4 Sonnet parse failures share one root cause: output token truncation. The model exhausts its output budget doing verbose inline computation in PLAN_STATE or SUB_N and never reaches the required structured fields. Fix is prompt hardening + optional max_tokens increase.

## Headline

**Single root cause: output truncation.** In all 4 failures, Sonnet spent its entire output token budget on inline computation (schedule tables, tour distances, variable elimination math, greedy vertex-checking), and the response was cut off before emitting any of the required structured fields (BEST_GUESS, UPDATED_PLAN_STATE, QUALITY_FORECAST, CONTINUE_FORECAST, DECISION or NEXT_SUB).

This is NOT a field-ordering error, NOT wrong JSON format, NOT extra fields confusing the parser. The fields are simply absent because the model never reached them.

---

## Per-Failure Table

| id | error | free-form section | content in section | truncated at | fields emitted | missing fields |
|---|---|---|---|---|---|---|
| `cjs_medium_seed1` | exec turn 2 parse failed | `SUB_1:` | Detailed machine-timeline schedule construction, row-by-row verification for both factories (~700 lines) | `"**MB3:** J4[43,"` (mid-table) | none after SUB_1 | BEST_GUESS, UPDATED_PLAN_STATE, QUALITY_FORECAST, CONTINUE_FORECAST, DECISION |
| `tsp_medium_seed1` | exec turn 2 parse failed | `SUB_1:` | Explicit Euclidean distance sums for 15+ candidate tours, 20 edges each | `"**Tour R:** 0→4→9→3→10→17→5→"` (mid-tour) | none after SUB_1 | BEST_GUESS, UPDATED_PLAN_STATE, QUALITY_FORECAST, CONTINUE_FORECAST, DECISION |
| `ve_medium_seed1` | exec turn 2 parse failed | `SUB_1:` | Step-by-step numerical variable elimination (factor products, sums over 16 variable combinations) | `"0.280272 + 0.632·(−0.052"` (mid-multiplication) | none after SUB_1 | BEST_GUESS, UPDATED_PLAN_STATE, QUALITY_FORECAST, CONTINUE_FORECAST, DECISION |
| `mwis_medium_seed1` | plan turn omitted NEXT_SUB | `PLAN_STATE:` | Greedy MWIS: vertex-by-vertex compatibility check across 120 vertices (~80 vertices checked) | `"Can V112=23 join?** Already in set.\n\n**"` (mid-analysis) | PLAN_STATE (partial) | NEXT_SUB |

**Key evidence of truncation (not just bad formatting):** all 4 response strings end mid-sentence or mid-number. There is zero text after the cut-off — no partial attempt at structured fields.

---

## Contrast: Why Steiner Passed

Steiner is a small-graph problem (~8 nodes). The model enumerated ~5 candidate trees (each 2–3 lines), checked them quickly, and produced all structured fields within ~300 words. Compact reasoning → fits in token budget → all fields present → strict parse succeeds.

The 4 failures are all computationally heavier problems (scheduling, distance geometry, Bayesian networks, combinatorial search over 120 nodes) where the model chose exhaustive inline computation rather than concise reasoning.

---

## Contributing Factor: LocalLLM has no max_tokens cap

`eval_harness/local_llm.py` calls the `llm` CLI with no `--max-tokens` option. The default for Claude via the `llm` Anthropic plugin is likely 4096 or 8192. For the CJS exec turn alone, the output appears to be several thousand tokens — potentially hitting the default ceiling even if the model's reasoning were appropriate.

---

## Recommendations

### Recommendation (a): Prompt hardening — PRIMARY fix

Two targeted wording changes in `harness/prompt.py`:

**Fix 1 — plan turn PLAN_STATE:**
```diff
- "- `PLAN_STATE`: free-form plan. We will quote this back to you verbatim on every later turn.\n"
+ "- `PLAN_STATE`: brief approach, ≤5 lines. No computation — reserve that for execution turns. We will quote this back to you verbatim.\n"
```

**Fix 2 — exec turn: add after the field list, before the example:**
```diff
- "All labelled fields must be emitted in the listed order. The `BEST_GUESS:` line must contain "
- "valid JSON, with no code fences and no extra text inside that field."
+ "All labelled fields must be emitted in the listed order. The `BEST_GUESS:` line must contain "
+ "valid JSON, with no code fences and no extra text inside that field.\n"
+ "CRITICAL: Keep SUB_N to ≤10 lines of high-level reasoning. If you run long, cut it short. "
+ "Failing to emit BEST_GUESS through DECISION causes parse failure — no score is recorded."
```

Also amend the SUB_N bullet itself:
```diff
- f"- `SUB_{turn_index - 1}`: reasoning/work for this subtask\n"
+ f"- `SUB_{turn_index - 1}`: ≤10 lines of reasoning for this subtask\n"
```

### Recommendation (b): Increase max_tokens in LocalLLM — SECONDARY safety net

Add `"--option", "max_tokens", "8192"` (or higher) to the llm cmd in `eval_harness/local_llm.py`. This is belt-and-suspenders: even if the model ignores the brevity constraint, it won't be silently truncated.

Option (a) is the load-bearing fix. Option (b) reduces residual risk.

---

## Is a fallback parser or parser relaxation defensible?

No — **not applicable here**. The missing fields are entirely absent (not malformed), so no regex relaxation rescues them. A fallback extractor for DECISION (e.g., infer from NEXT_SUB presence) doesn't help because BEST_GUESS and QUALITY_FORECAST are also missing. Ayu's "no fallback" constraint is not at risk from this fix.

---

If prompt hardening (a) + max_tokens increase (b) applied together:
