## Plugin Source Evidence

### llm-anthropic v0.24 (`llm_anthropic.py`)
```python
# ClaudeOptions (line 282)
max_tokens: int | None = Field(default=None)

# _Shared class (line 484)
default_max_tokens = 4096  # class-level fallback

# claude-sonnet-4-6 registration (line 264)
ClaudeMessages("claude-sonnet-4-6", ..., default_max_tokens=8192)

# build_kwargs (line 766-775)

    max_tokens = prompt.options.max_tokens

```
- Option key `max_tokens` is **always** passed to API (no silent-accept risk)
- 128k beta header added only if `max_tokens > 64000 AND NOT supports_adaptive_thinking` — Sonnet 4.6 has `supports_adaptive_thinking=True` so 64k is the practical ceiling without extra headers

### llm-gemini v0.30 (`llm_gemini.py`)

```python
    ...

```
- **`--option max_tokens` → hard validation error**: `Extra inputs are not permitted` (Pydantic reject)
- No default set → Gemini model uses its own internal default (~8192 in practice)

### llm core / OpenAI plugin (`openai_models.py`)

## PREDICTION CLAIM

With `--option max_tokens 40000` applied to `claude-sonnet-4.6` runs, Sonnet truncation failures **will drop from 4/6 to 0/6**.

**Reasoning:** All 4 failures share one root cause (output truncation at ~8192 token budget). The model exhausted its budget doing inline computation before emitting structured fields. Raising to 40000 tokens (5× the current default, comfortably under the 64k ceiling) eliminates the budget constraint. The structured fields are present in the model's intended output — they're simply unreachable at 8k.

### NOTES
