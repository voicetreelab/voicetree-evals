# Portfolio v2 — raw-string conversation loop + LLM-as-extractor

# Why this exists
Tara (portfolio_spike regex parser) and Uma (structured-output with Pydantic/Gemini schema) both died at turn 1 from protocol-layer format issues. Zero execution data captured.

The user has instructed a clean architecture: **no parsing during the run**, just raw-string conversation append; use a post-hoc LLM extractor to pull the final candidate answer and session metrics from the transcript. This eliminates the entire parser-failure class.

**Do NOT modify `hch/portfolio_spike/` — leave it intact as reference.** Create a new folder `hch/portfolio_v2/`.

# Architecture

## During the run: dumb conversation loop
```
conversation_history = [system_prompt]
for turn in range(MAX_TURNS):
    if wall_elapsed > 1800: break
    turn_context = f"Turn {turn}. Elapsed: {wall_elapsed}s. Remaining: {1800-wall_elapsed}s. Continue with your next subtask. Emit your current best candidate answer for any problem you've updated (JSON, e.g. P1_ANSWER: {...})."
    response = gemini.send(conversation_history + [turn_context])
    conversation_history.append(response)
    # NO PARSING. NO VALIDATION. Just save and loop.
```

- Fixed cap: `MAX_TURNS=10`, `WALL_BUDGET_S=1800`
- No model-side stop-signal trust. Turn cap + wall cap only.
- Optional loose regex sniff for an emergency `[[TERMINATE]]` marker — but default behavior: run to budget.

## System prompt contents (keep compact)
- All 4 problem descriptions (P1 jobshop, P2 Steiner×coloring, P3 TSP-20, P4 graph coloring)
- Baseline values + optimal values per problem (no per-problem value cap mystery)
- Economic objective: `Σ value_cap × headroom - cost_per_second × wall_s`, cost_per_second=0.05, value_caps=[50,60,20,100]
- Instruction to emit structured candidate answers in JSON when the model updates them (e.g. `P1_ANSWER: {...}`) — but this is a **hint, not enforced**. Free-form elsewhere.
- Instruction to decompose, revise plan across turns, add new subtasks when learning — plain English, no schema.

## Post-run: LLM-as-extractor (`extract.py`)
**Extractor model:** Gemini 2.5 Flash (cheap, fast, already supported by existing client). Fall back to Gemini 3 Pro if Flash is unreliable.

**Per-problem final-answer extraction (4 calls):**
```
prompt = f"""Below is a transcript of an AI attempting problem {problem_id} ({short_desc}).
Extract the AI's FINAL candidate answer — the latest best attempt.
If no candidate answer is present, return {{"missing": true}}.
Expected schema for {problem_id}: {schema_hint}
Output ONLY valid JSON: {{"answer": <extracted>, "notes": "<1-line summary of what was extracted>"}}

Transcript:
<<<
{full_transcript}
>>>
"""
```

Feed extracted `answer` into the **existing per-problem verifier** (`verify_schedule`, `verify_tour`, `verify_graph_coloring_answer`, `verify_steiner_answer` from portfolio_spike — just import them). Verifier returns feasible/infeasible + verified_score. Compute value_captured from verified_score.

**Session metrics extraction (1 more call):**
```
prompt = f"""Transcript below. Extract:
- turn_count: number of distinct turns where the model produced content
- subtasks_executed: distinct subtasks the model actually worked on (not just planned)
- problems_attempted: which of [P1, P2, P3, P4] the model generated any candidate answer for
- plan_evolution_summary: one sentence — did the model add/revise subtasks mid-run?
- declared_stop_reason: did the model ever say it wanted to stop, or did it just run to budget?
Output JSON: {{"turn_count": N, "subtasks_executed": N, "problems_attempted": [...], "plan_evolution_summary": "...", "declared_stop_reason": "..."}}
"""
```

These are the observability signals. If extraction quality is low, note it in the progress node but don't iterate.
